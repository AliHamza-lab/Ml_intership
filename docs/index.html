Abstract

In digital publishing operations, content teams struggle to decide which of their thousands of existing pages should be prioritized for updates when search visibility begins to decline. This paper frames a public-safe content triage problem using an anonymized dataset of 30,000 production pages. We construct a transparent baseline rule scoring system and compare it to three learned classifiers under a client-holdout validation design to prevent data leakage. The Random Forest classifier achieves the best performance with a Test ROC AUC of 0.747 and Average Precision of 0.610, delivering a 2.8x lift in Precision@50 (0.680 vs. 0.240) over the baseline rules. The final output is formatted as a prioritized queue and action playbook to support editorial decisions under resource constraints.

Introduction & Problem Statement

For large-scale digital publishers, refreshing existing content is one of the most efficient growth and maintenance levers. However, editorial budgets and writer hours are strictly limited, making it impossible to manually audit every page. Reviewing high-traffic pages that show signs of performance decay (fading search visibility, low click-through rate, or dropping engagement) is critical.

The core decision this supports is editorial triage: which pages should be sent to a reviewer for updates? Instead of letting pages decay or guess which ones are slipping, editorial operations need a rational, data-driven priority queue.

Unit of Analysis

Individual page / content item. It represents the level at which editorial changes are made.

Outputs

A final priority score (0-100), suggested review actions (e.g. metadata vs depth), and a confidence tier.

Editor Action

Review the prioritized page, compare it against recent search queries, and update or expand the content.

Cost of Wrong Calls

False Positive: Editorial hours are wasted rewrite-checking healthy pages. False Negative: High-value pages slip in rankings unnoticed, losing traffic.

Data & Safety

The analysis runs entirely on the anonymized starter slice (data/raw/content_refresh_anonymized.csv). To maintain absolute data safety and comply with privacy rules, all sensitive fields have been removed:

  • No client names, domains, or page URLs.
  • No page titles or raw search queries.
  • Only hashed IDs (content_id and client_id) and numeric metrics remain.

Exclusion Decisions & Rationale

To ensure the model learns robust patterns, we apply a strict prep pipeline:

Impressions > 0

Exclude pages with zero impressions in the last 90 days. If a page has no search visibility, there is no performance trend to evaluate or save through a refresh.

Page Age >= 90 Days

Exclude pages published for less than 90 days. Fresh pages are in their initial indexation/ranking discovery phase, where volatility is normal and a refresh review is premature.

Methodology

Label Definition

The target label is defined as: is_declining_label = (trend_direction == 'down'). This indicates search visibility loss over recent comparison windows.

Leakage Prevention Check

Because the target is derived from the search trend, both trend_direction and trend_pct are strictly excluded from the feature matrix. Including them would represent a critical data leak, inflating model metrics artificially.

Grouped Validation Split

Rather than a standard row-wise split, we implement a Grouped Client-Holdout split (80% train / 20% test based on client_id). This guarantees that no individual client's pages appear in both training and test sets. It mimics real-world deployment where a model must generalize to entirely new web properties.

Baseline Rule Score

To assess model value, we construct a transparent baseline heuristic rules score (baseline_refresh_score): 

baseline_score = (
    0.40 * visibility_score + 
    0.30 * freshness_risk_score + 
    0.25 * position_opportunity_score + 
    0.05 * depth_gap_score
)

This rule captures the standard editorial instincts: prioritizing pages that are stale (freshness), highly visible (visibility), ranking just outside the top-3 (position opportunity), and thin (depth gap).

Results (Model vs. Baseline)

We train Logistic Regression, Decision Tree, and Random Forest models using the client-holdout split. The models are compared against the baseline rules score on the same test set:

Key Observations

The Random Forest classifier shows superior performance. Crucially, on our select triage metric (Precision@50), the baseline rules score yields only 0.240, while the Random Forest classifier yields 0.680. This represents a 2.83x lift in triage accuracy. An editor auditing the top 50 rows would find 34 actual declining pages using the model, compared to only 12 using the baseline rule.

Limitations & Honest Framing

Transparent Research Disclosures

We present these findings with the following constraints to ensure honest interpretation:

  • Static Historical Window: The results are measured on a static labeled export of historical data. Because search engine algorithms and keyword volumes change dynamically, we cannot guarantee that these specific predictive weights remain permanent.
  • Correlation vs. Causality: The model maps observable correlation signals. We do not claim that high impressions cause decline, or that performing a refresh guarantees traffic recovery. The score serves strictly as an operational triage filter.
  • Lack of Longitudinal Field Experiment: The performance lift is validated on offline historical test splits. A true measure of editorial ROI requires a live, randomized A/B trial of suggested updates vs. untreated controls.

Recommendations & Action Playbook

The priority queue score (final_refresh_score) blends the Random Forest prediction probability (70%) with the normalized baseline rules score (30%). This scores each page from 0 to 100.

We map these scores into targeted editorial actions, providing a playbook for reviewers:

refresh_and_review_ctr

Trigger: High visibility and high decline risk, but low CTR. Action: Editors should optimize search snippets, titles, metadata, and headings to improve CTR.

refresh_and_review_engagement

Trigger: High click volumes and decline risk, but low scroll/engagement. Action: Editors should improve article introduction, structure, layout, and match it to user intent.

expand_and_refresh

Trigger: Stale visible pages with low word count. Action: Add details, expand sections, and enrich content depth.

refresh

Trigger: Stale visible pages with general decline risk. Action: Update stale statistics, links, and text.

Reproducibility & Code

All experimental notebooks, feature preparation scripts, training runs, and evaluations are fully open-source and structured for easy execution.

Step-by-Step Rerun Instructions

Execute the following commands from a clean terminal to clone and reproduce the pipeline outputs:

# 1. Clone the repository
git clone https://github.com/AliHamza-lab/Ml_intership.git
cd Ml_intership

# 2. Setup the virtual environment and install dependencies
python -m venv .venv
.venv\\Scripts\\activate   # For Windows
# source .venv/bin/activate  # For Unix/macOS
pip install -r requirements.txt

# 3. Run the entire pipeline
python scripts/run_all.py

The pipeline takes approximately 1 minute to complete and regenerates all prediction CSV files and model results JSON metadata.
